Key Concepts (HTTP Essentials)

Core

URL
Uniform Resource Locator: scheme + host + path + query (and optional fragment). Example: https://example.com/products?id=42.
HTTP Methods
GET (read), POST (create), PUT/PATCH (update), DELETE (remove), plus HEAD, OPTIONS. Methods should be used semantically.
Status Codes
2xx success, 3xx redirection, 4xx client errors, 5xx server errors. Example: 200 OK, 301 Moved Permanently, 404 Not Found, 500 Internal Server Error.
Headers
Key–value metadata that shapes behavior (e.g., Content-Type, Accept, Cache-Control, Set-Cookie, Authorization).
Body & Content Types
HTTP carries HTML, JSON, images, video, etc. Expressed via Content-Type (text/html, application/json, image/png).

Performance & Reliability

Caching
Cache-Control: max-age, revalidation with ETag/If-None-Match, and Last-Modified. Reduces latency and bandwidth.
Connections & Versions
HTTP/1.1 adds persistent connections and chunked transfer; HTTP/2 multiplexes streams; HTTP/3 runs over QUIC for better loss recovery.
Content Negotiation
Clients express preferences with Accept, Accept-Language, Accept-Encoding; servers choose the best representation.

Security

TLS / HTTPS
Encrypts transport for confidentiality and integrity. “HTTPS is now the norm,” improving trust and enabling modern features.
Cookies
State on the client—use HttpOnly, Secure, SameSite. Set by Set-Cookie, sent back via Cookie.
Auth
Mechanisms include Basic (over TLS only), Bearer tokens (Authorization), and session cookies. Always prefer HTTPS.
CORS & Same-Origin
Browsers restrict cross-origin reads. Servers can opt-in via Access-Control-Allow-Origin and related headers.
HSTS
Strict-Transport-Security forces HTTPS for a period, mitigating protocol downgrades and cookie leakage.

Modern Patterns

REST & Resources
Model data as resources with stable URLs; use proper methods and status codes; keep representations cacheable when possible.
APIs & JSON
Most public APIs speak HTTP + JSON. Prefer explicit versioning and clear error formats with application/problem+json when applicable.
Idempotency & Safety
GET/HEAD are safe (no state change). PUT/DELETE are idempotent (repeatable). Design clients/servers to honor these guarantees.
CDNs
Push content closer to users, honor cache headers, and terminate TLS near the edge. Critical for global performance.

Examples

GET /api/items/42 HTTP/1.1
Host: example.com
Accept: application/json

HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: max-age=600
ETag: "v2-1a7"

{"id":42,"name":"widget","price":19.99}

POST /login HTTP/1.1
Host: example.com
Content-Type: application/json

{"email":"you@example.com","password":"secret"}

Server sets a secure session cookie:

HTTP/1.1 200 OK
Set-Cookie: sid=abc123; HttpOnly; Secure; SameSite=Lax
“HTTP/1.1’s persistent connections reduced overhead and helped the web scale.”
“HTTP/2 multiplexing cut head-of-line blocking and improved page load times.”
“HTTP/3 over QUIC increases resilience on lossy, mobile networks.”
