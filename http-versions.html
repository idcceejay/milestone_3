HTTP Versions – From 0.9 to 3

HTTP evolved with the Web—from a single-method text protocol to today’s encrypted, mobile-friendly transport over QUIC. Below is a comparison matrix, per-version notes, and the full essay that informs this page.

HTTP Feature Matrix
Version Era / Standardization Transport Key Features Performance & Notes
0.9 Early 1990s (CERN) TCP Single GET; raw HTML; no headers Ultra-simple and stateless; enabled the first web pages.
1.0 1996 (first official spec) TCP Headers, MIME types, status codes (e.g., 404) Many TCP connections per page; higher latency.
1.1 1997 (dominant for ~20 years) TCP Persistent connections, caching, chunked transfer Faster loads via keep-alive; still suffered HoL blocking per connection.
2 2015 TCP Multiplexing, binary framing, header compression; server push Major latency reduction; simultaneous streams on one TCP connection; widely adopted.
3 Standardized 2022 QUIC over UDP Built-in encryption, faster handshakes (0-RTT), better loss recovery Mobile-friendly and resilient; adoption accelerating across CDNs and major sites.

HTTP/0.9 – The Seed of the Web

  • Only GET, no headers—returned raw HTML; simple by design for early documents.
  • Established a browser–server pattern that scaled beyond academic use.

HTTP/1.0 – Rich Content Arrives

  • Introduced headers, MIME types, and status codes—enabling images and multimedia.
  • Latency rose because each page often opened multiple TCP connections.

HTTP/1.1 – The Dot-Com Workhorse

  • Persistent connections (keep-alive), chunked transfer, and improved caching.
  • Dominated for nearly two decades; enabled large-scale commercial web apps.

HTTP/2 – Parallelism on a Single Connection

  • Binary framing + multiplexing removes request blocking on a single TCP connection.
  • Header compression cuts overhead; optional server push can pre-send assets.

HTTP/3 – QUIC for the Mobile Internet

  • Runs over QUIC (UDP) with built-in encryption and faster handshakes (including 0-RTT).
  • More resilient to packet loss and network changes (Wi-Fi ↔ LTE handoffs).

Developer Takeaways

  1. Prefer HTTPS everywhere. Security & platform features assume TLS.
  2. Design for HTTP/2/3. Parallel requests are fine; still minimize bytes & round trips.
  3. Cache correctly. Use explicit freshness and validators to cut latency.
  4. Measure user-perceived speed. Track LCP/TTFB to validate protocol gains.
